# Implement Feature Skill

## Token Efficiency (Default)

- Default mode: `compact`.
- Start with minimal context: touched module files only.
- First-pass read budget: max 8 file reads before first edit.
- Expand scope only on blocker or failing validation.
- Progress updates: delta only, no repeated plan blocks.

## When to Use

- Implement a feature after requirement analysis is already completed with `read-feature`
- Deliver end-to-end feature changes in Flutter code: model, UI, and controller behavior
- Keep implementation aligned with project architecture in `belotable/lib/`
- Add validation through unit tests and `integration_test`

## Expected Input

Input file must be a markdown analysis generated from `read-feature` and include:

- `Summary`
- `Use Cases`
- `Data Model` table
- `User Interface` table
- `Acceptance Criteria`
- Optional `Risks / Dependencies`

If missing sections, stop and ask user to complete analysis first.

## Architecture Constraints (Repository-Specific)

- `belotable/lib/data/`: persistence, DTOs, repositories, adapters
- `belotable/lib/domain/`: business rules, entities, use-case logic
- `belotable/lib/presentation/`: widgets, screens, interaction flow
- `belotable/lib/gen/`: generated code only, do not edit manually

Apply MVC intent without breaking existing layering:

- Model: `data/` + `domain/`
- View: `presentation/`
- Controller logic: orchestrating interaction logic in current project pattern (existing controllers/view-model/state handlers). Reuse existing pattern before introducing new state management.

## Implementation Workflow

1. Parse Analysis and Build Change Matrix
   - Read analysis markdown.
   - Build matrix linking each acceptance criterion to:
     - model changes
     - controller logic changes
     - UI changes
     - tests to create/update

2. Validate Scope and Branching Decisions
   - If `Data Model` section has `NEW` or `MODIFIED` rows: implement model/repository/domain updates.
   - If `User Interface` section has `NEW`, `MODIFIED`, or `REMOVED` rows: implement screen/widget updates.
   - If a criterion needs orchestration (state transitions, validation, sequencing): implement/update controller logic.
   - If `Deprecated Behaviors` section is present: ask user decision per item before coding (`remove now`, `soft-deprecate`, or `postpone`).
   - If table entries conflict with acceptance criteria: ask user for priority before coding.

3. Implement Model Layer Changes
   - Add or update domain entities/value objects in `belotable/lib/domain/`.
   - Add or update data models/mappers/repositories in `belotable/lib/data/`.
   - Keep serialization and conversion paths consistent.
   - Preserve backward compatibility when behavior marked `MODIFIED` rather than `REMOVED`.

4. Implement Controller Logic
   - Locate existing interaction orchestration type in touched feature area.
   - Extend existing controller/view-model/state handler instead of creating parallel pattern.
   - If no orchestration pattern exists in touched module, ask user before creating a new controller/view-model type.
   - Encode business decision points explicitly (guard clauses, state transitions, error handling).
   - Keep algorithmic logic testable outside widgets where possible.

5. Implement View Layer Changes
   - Update or create screens/widgets under `belotable/lib/presentation/`.
   - Wire view actions to controller behavior.
   - Ensure changed UI still matches acceptance criteria wording.
   - Keep accessibility and responsive behavior coherent with existing app patterns.
   - UI text and labels must be in French as per project localization standards. Use special characters and accents as needed.

6. Add/Update Unit Tests
   - Add tests in `belotable/test/` for:
     - domain rules
     - controller logic
     - data mapping/repository behavior where feasible
   - For each acceptance criterion, ensure at least one unit assertion exists or document why integration-only.

7. Add/Update Integration Tests
   - Add flow coverage in `belotable/integration_test/` (flows, pages, robots, helpers).
   - Ensure end-to-end user journeys cover new/modified behaviors.
   - Integration tests are optional when unit tests fully cover behavior and no user flow changed.
   - Register or update suite composition in `belotable/integration_test/all_tests.dart` when needed.

8. Run Quality Gates
   - Run static analysis and tests relevant to touched code.
   - Required validation before finalizing:
     - `fvm flutter analyze`
     - `fvm flutter test`
     - `fvm flutter test integration_test/all_tests.dart -d windows` (or documented subset if full run is too expensive)

9. Produce Delivery Summary
   - Report what was implemented by layer (Model, Controller, View).
   - List test files added/updated and criteria covered.
   - List deferred items, assumptions, and blockers.

## Completion Criteria

Feature implementation is complete only when all are true:

- Every acceptance criterion is mapped to code and tests.
- No unresolved architecture violations (layer leakage between `presentation`, `domain`, `data`).
- Unit tests are added or updated for behavioral changes.
- Integration tests are added when user flow changes; otherwise omission must be explicitly justified.
- Analysis and test commands pass, or failures are explicitly reported with cause.

## Failure Handling

- If analysis file is ambiguous, ask targeted questions before implementation.
- If existing project pattern is unclear, inspect nearby feature modules and follow dominant style.
- If requirement cannot be implemented without breaking architecture, stop and propose a minimal compliant design alternative.

## Chat Output Contract

Return:

- 1 short implementation summary
- validation command results
- open questions or follow-up tasks

Compact output format (default):
- `Summary:` <=2 lines
- `Changed:` paths only
- `Validation:` command + pass/fail only
- `Open:` <=3 bullets
